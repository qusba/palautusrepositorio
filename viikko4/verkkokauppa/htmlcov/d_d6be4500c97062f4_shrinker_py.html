40def sort_key(buffer): 

41 """Returns a sort key such that "simpler" buffers are smaller than 

42 "more complicated" ones. 

43 

44 We define sort_key so that x is simpler than y if x is shorter than y or if 

45 they have the same length and x < y lexicographically. This is called the 

46 shortlex order. 

47 

48 The reason for using the shortlex order is: 

49 

50 1. If x is shorter than y then that means we had to make fewer decisions 

51 in constructing the test case when we ran x than we did when we ran y. 

52 2. If x is the same length as y then replacing a byte with a lower byte 

53 corresponds to reducing the value of an integer we drew with draw_bits 

54 towards zero. 

55 3. We want a total order, and given (2) the natural choices for things of 

56 the same size are either the lexicographic or colexicographic orders 

57 (the latter being the lexicographic order of the reverse of the string). 

58 Because values drawn early in generation potentially get used in more 

59 places they potentially have a more significant impact on the final 

60 result, so it makes sense to prioritise reducing earlier values over 

61 later ones. This makes the lexicographic order the more natural choice. 

62 """ 

63 return (len(buffer), buffer)

112class Shrinker(object): 

113 """A shrinker is a child object of a ConjectureRunner which is designed to 

114 manage the associated state of a particular shrink problem. That is, we 

115 have some initial ConjectureData object and some property of interest 

116 that it satisfies, and we want to find a ConjectureData object with a 

117 shortlex (see sort_key above) smaller buffer that exhibits the same 

118 property. 

119 

120 Currently the only property of interest we use is that the status is 

121 INTERESTING and the interesting_origin takes on some fixed value, but we 

122 may potentially be interested in other use cases later. 

123 However we assume that data with a status < VALID never satisfies the predicate. 

124 

125 The shrinker keeps track of a value shrink_target which represents the 

126 current best known ConjectureData object satisfying the predicate. 

127 It refines this value by repeatedly running *shrink passes*, which are 

128 methods that perform a series of transformations to the current shrink_target 

129 and evaluate the underlying test function to find new ConjectureData 

130 objects. If any of these satisfy the predicate, the shrink_target 

131 is updated automatically. Shrinking runs until no shrink pass can 

132 improve the shrink_target, at which point it stops. It may also be 

133 terminated if the underlying engine throws RunIsComplete, but that 

134 is handled by the calling code rather than the Shrinker. 

135 

136 ======================= 

137 Designing Shrink Passes 

138 ======================= 

139 

140 Generally a shrink pass is just any function that calls 

141 cached_test_function and/or incorporate_new_buffer a number of times, 

142 but there are a couple of useful things to bear in mind. 

143 

144 A shrink pass *makes progress* if running it changes self.shrink_target 

145 (i.e. it tries a shortlex smaller ConjectureData object satisfying 

146 the predicate). The desired end state of shrinking is to find a 

147 value such that no shrink pass can make progress, i.e. that we 

148 are at a local minimum for each shrink pass. 

149 

150 In aid of this goal, the main invariant that a shrink pass much 

151 satisfy is that whether it makes progress must be deterministic. 

152 It is fine (encouraged even) for the specific progress it makes 

153 to be non-deterministic, but if you run a shrink pass, it makes 

154 no progress, and then you immediately run it again, it should 

155 never succeed on the second time. This allows us to stop as soon 

156 as we have run each shrink pass and seen no progress on any of 

157 them. 

158 

159 This means that e.g. it's fine to try each of N deletions 

160 or replacements in a random order, but it's not OK to try N random 

161 deletions (unless you have already shrunk at least once, though we 

162 don't currently take advantage of this loophole). 

163 

164 Shrink passes need to be written so as to be robust against 

165 change in the underlying shrink target. It is generally safe 

166 to assume that the shrink target does not change prior to the 

167 point of first modification - e.g. if you change no bytes at 

168 index ``i``, all examples whose start is ``<= i`` still exist, 

169 as do all blocks, and the data object is still of length 

170 ``>= i + 1``. This can only be violated by bad user code which 

171 relies on an external source of non-determinism. 

172 

173 When the underlying shrink_target changes, shrink 

174 passes should not run substantially more test_function calls 

175 on success than they do on failure. Say, no more than a constant 

176 factor more. In particular shrink passes should not iterate to a 

177 fixed point. 

178 

179 This means that shrink passes are often written with loops that 

180 are carefully designed to do the right thing in the case that no 

181 shrinks occurred and try to adapt to any changes to do a reasonable 

182 job. e.g. say we wanted to write a shrink pass that tried deleting 

183 each individual byte (this isn't an especially good choice, 

184 but it leads to a simple illustrative example), we might do it 

185 by iterating over the buffer like so: 

186 

187 .. code-block:: python 

188 

189 i = 0 

190 while i < len(self.shrink_target.buffer): 

191 if not self.incorporate_new_buffer( 

192 self.shrink_target.buffer[: i] + 

193 self.shrink_target.buffer[i + 1 :] 

194 ): 

195 i += 1 

196 

197 The reason for writing the loop this way is that i is always a 

198 valid index into the current buffer, even if the current buffer 

199 changes as a result of our actions. When the buffer changes, 

200 we leave the index where it is rather than restarting from the 

201 beginning, and carry on. This means that the number of steps we 

202 run in this case is always bounded above by the number of steps 

203 we would run if nothing works. 

204 

205 Another thing to bear in mind about shrink pass design is that 

206 they should prioritise *progress*. If you have N operations that 

207 you need to run, you should try to order them in such a way as 

208 to avoid stalling, where you have long periods of test function 

209 invocations where no shrinks happen. This is bad because whenever 

210 we shrink we reduce the amount of work the shrinker has to do 

211 in future, and often speed up the test function, so we ideally 

212 wanted those shrinks to happen much earlier in the process. 

213 

214 Sometimes stalls are inevitable of course - e.g. if the pass 

215 makes no progress, then the entire thing is just one long stall, 

216 but it's helpful to design it so that stalls are less likely 

217 in typical behaviour. 

218 

219 The two easiest ways to do this are: 

220 

221 * Just run the N steps in random order. As long as a 

222 reasonably large proportion of the operations suceed, this 

223 guarantees the expected stall length is quite short. The 

224 book keeping for making sure this does the right thing when 

225 it succeeds can be quite annoying. 

226 * When you have any sort of nested loop, loop in such a way 

227 that both loop variables change each time. This prevents 

228 stalls which occur when one particular value for the outer 

229 loop is impossible to make progress on, rendering the entire 

230 inner loop into a stall. 

231 

232 However, although progress is good, too much progress can be 

233 a bad sign! If you're *only* seeing successful reductions, 

234 that's probably a sign that you are making changes that are 

235 too timid. Two useful things to offset this: 

236 

237 * It's worth writing shrink passes which are *adaptive*, in 

238 the sense that when operations seem to be working really 

239 well we try to bundle multiple of them together. This can 

240 often be used to turn what would be O(m) successful calls 

241 into O(log(m)). 

242 * It's often worth trying one or two special minimal values 

243 before trying anything more fine grained (e.g. replacing 

244 the whole thing with zero). 

245 

246 """

513 def fixate_shrink_passes(self, passes): 

514 """Run steps from each pass in ``passes`` until the current shrink target 

515 is a fixed point of all of them.""" 

516 passes = list(map(self.shrink_pass, passes)) 

517 

518 any_ran = True 

519 while any_ran: 

520 any_ran = False 

521 

522 # We run remove_discarded after every step to do cleanup 

523 # keeping track of whether that actually works. Either there is 

524 # no discarded data and it is basically free, or it reliably works 

525 # and deletes data, or it doesn't work. In that latter case we turn 

526 # it off for the rest of this loop through the passes, but will 

527 # try again once all of the passes have been run. 

528 can_discard = self.remove_discarded() 

529 

530 successful_passes = set() 

531 

532 for sp in passes: 

533 # We run each pass until it has failed a certain number 

534 # of times, where a "failure" is any step where it made 

535 # at least one call and did not result in a shrink. 

536 # This gives passes which work reasonably often more of 

537 # chance to run. 

538 failures = 0 

539 successes = 0 

540 

541 # The choice of 3 is fairly arbitrary and was hand tuned 

542 # to some particular examples. It is very unlikely that 

543 # is the best choice in general, but it's not an 

544 # unreasonable choice: Making it smaller than this would 

545 # give too high a chance of an otherwise very worthwhile 

546 # pass getting screened out too early if it got unlucky, 

547 # and making it much larger than this would result in us 

548 # spending too much time on bad passes. 

549 max_failures = 3 

550 

551 while failures < max_failures: 

552 prev_calls = self.calls 

553 prev = self.shrink_target 

554 if sp.step(): 

555 any_ran = True 

556 else: 

557 break 

558 if prev_calls != self.calls: 

559 if can_discard: 

560 can_discard = self.remove_discarded() 

561 if prev is self.shrink_target: 

562 failures += 1 

563 else: 

564 successes += 1 

565 if successes > 0: 

566 successful_passes.add(sp) 

567 

568 # If only some of our shrink passes are doing anything useful 

569 # then run all of those to a fixed point before running the 

570 # full set. This is particularly important when an emergency 

571 # shrink pass unlocks some non-emergency ones and it suddenly 

572 # becomes very expensive to find a bunch of small changes. 

573 if 0 < len(successful_passes) < len(passes): 

574 self.fixate_shrink_passes(successful_passes) 

575 

576 for sp in passes: 

577 sp.fixed_point_at = self.shrink_target

609 def pass_to_descendant(self, chooser): 

610 """Attempt to replace each example with a descendant example. 

611 

612 This is designed to deal with strategies that call themselves 

613 recursively. For example, suppose we had: 

614 

615 binary_tree = st.deferred( 

616 lambda: st.one_of( 

617 st.integers(), st.tuples(binary_tree, binary_tree))) 

618 

619 This pass guarantees that we can replace any binary tree with one of 

620 its subtrees - each of those will create an interval that the parent 

621 could validly be replaced with, and this pass will try doing that. 

622 

623 This is pretty expensive - it takes O(len(intervals)^2) - so we run it 

624 late in the process when we've got the number of intervals as far down 

625 as possible. 

626 """ 

627 

628 label = chooser.choose( 

629 self.distinct_labels, lambda l: len(self.examples_by_label[l]) >= 2 

630 ) 

631 

632 ls = self.examples_by_label[label] 

633 

634 i = chooser.choose(hrange(len(ls) - 1)) 

635 

636 ancestor = ls[i] 

637 

638 @self.cached(label, i) 

639 def descendants(): 

640 lo = i + 1 

641 hi = len(ls) 

642 while lo + 1 < hi: 

643 mid = (lo + hi) // 2 

644 if ls[mid].start >= ancestor.end: 

645 hi = mid 

646 else: 

647 lo = mid 

648 return ls[i + 1 : hi] 

649 

650 descendant = chooser.choose(descendants, lambda ex: ex.length > 0) 

651 

652 self.incorporate_new_buffer( 

653 self.buffer[: ancestor.start] 

654 + self.buffer[descendant.start : descendant.end] 

655 + self.buffer[ancestor.end :] 

656 )

658 def lower_common_block_offset(self): 

659 """Sometimes we find ourselves in a situation where changes to one part 

660 of the byte stream unlock changes to other parts. Sometimes this is 

661 good, but sometimes this can cause us to exhibit exponential slow 

662 downs! 

663 

664 e.g. suppose we had the following: 

665 

666 m = draw(integers(min_value=0)) 

667 n = draw(integers(min_value=0)) 

668 assert abs(m - n) > 1 

669 

670 If this fails then we'll end up with a loop where on each iteration we 

671 reduce each of m and n by 2 - m can't go lower because of n, then n 

672 can't go lower because of m. 

673 

674 This will take us O(m) iterations to complete, which is exponential in 

675 the data size, as we gradually zig zag our way towards zero. 

676 

677 This can only happen if we're failing to reduce the size of the byte 

678 stream: The number of iterations that reduce the length of the byte 

679 stream is bounded by that length. 

680 

681 So what we do is this: We keep track of which blocks are changing, and 

682 then if there's some non-zero common offset to them we try and minimize 

683 them all at once by lowering that offset. 

684 

685 This may not work, and it definitely won't get us out of all possible 

686 exponential slow downs (an example of where it doesn't is where the 

687 shape of the blocks changes as a result of this bouncing behaviour), 

688 but it fails fast when it doesn't work and gets us out of a really 

689 nastily slow case when it does. 

690 """ 

691 if len(self.__changed_blocks) <= 1: 

692 return 

693 

694 current = self.shrink_target 

695 

696 blocked = [current.buffer[u:v] for u, v in self.all_block_bounds()] 

697 

698 changed = [ 

699 i 

700 for i in sorted(self.__changed_blocks) 

701 if not self.shrink_target.blocks[i].trivial 

702 ] 

703 

704 if not changed: 

705 return 

706 

707 ints = [int_from_bytes(blocked[i]) for i in changed] 

708 offset = min(ints) 

709 assert offset > 0 

710 

711 for i in hrange(len(ints)): 

712 ints[i] -= offset 

713 

714 def reoffset(o): 

715 new_blocks = list(blocked) 

716 for i, v in zip(changed, ints): 

717 new_blocks[i] = int_to_bytes(v + o, len(blocked[i])) 

718 return self.incorporate_new_buffer(hbytes().join(new_blocks))

731 def __changed_blocks(self): 

732 if self.__last_checked_changed_at is not self.shrink_target: 

733 prev_target = self.__last_checked_changed_at 

734 new_target = self.shrink_target 

735 assert prev_target is not new_target

737 new = new_target.buffer 

738 assert sort_key(new) < sort_key(prev) 

739 

740 if ( 

741 len(new_target.blocks) != len(prev_target.blocks) 

742 or new_target.blocks.endpoints != prev_target.blocks.endpoints 

743 ): 

744 self.__all_changed_blocks = set() 

745 else: 

746 blocks = new_target.blocks 

747

749 # by checking if the tail past this point has been modified. 

750 last_changed = binary_search( 

751 0, 

752 len(blocks), 

753 lambda i: prev[blocks.start(i) :] != new[blocks.start(i) :], 

754 ) 

755 

756 # Index of the first block whose contents have been changed, 

757 # because we know that this predicate is true for zero (because 

758 # the prefix from the start is empty), so the result must be True 

759 # for the bytes from the start of this block and False for the 

760 # bytes from the end, hence the change is in this block. 

761 first_changed = binary_search(

764 lambda i: prev[: blocks.start(i)] == new[: blocks.start(i)],

768 # check if any specific block values have changed.

788 def try_shrinking_blocks(self, blocks, b): 

789 """Attempts to replace each block in the blocks list with b. Returns 

790 True if it succeeded (which may include some additional modifications 

791 to shrink_target). 

792 

793 In current usage it is expected that each of the blocks currently have 

794 the same value, although this is not essential. Note that b must be 

795 < the block at min(blocks) or this is not a valid shrink. 

796 

797 This method will attempt to do some small amount of work to delete data 

798 that occurs after the end of the blocks. This is useful for cases where 

799 there is some size dependency on the value of a block. 

800 """ 

801 initial_attempt = bytearray(self.shrink_target.buffer) 

802 for i, block in enumerate(blocks): 

803 if block >= len(self.blocks): 

804 blocks = blocks[:i] 

805 break 

806 u, v = self.blocks[block].bounds 

807 n = min(self.blocks[block].length, len(b)) 

808 initial_attempt[v - n : v] = b[-n:] 

809 

810 if not blocks: 

811 return False 

812 

813 start = self.shrink_target.blocks[blocks[0]].start 

814 end = self.shrink_target.blocks[blocks[-1]].end 

815 

816 initial_data = self.cached_test_function(initial_attempt) 

817 

818 if initial_data.status == Status.INTERESTING: 

819 self.lower_common_block_offset() 

820 return initial_data is self.shrink_target 

821 

822 # If this produced something completely invalid we ditch it 

823 # here rather than trying to persevere. 

824 if initial_data.status < Status.VALID: 

825 return False 

826 

827 # We've shrunk inside our group of blocks, so we have no way to 

828 # continue. (This only happens when shrinking more than one block at 

829 # a time). 

830 if len(initial_data.buffer) < v: 

831 return False 

832 

833 lost_data = len(self.shrink_target.buffer) - len(initial_data.buffer) 

834 

835 # If this did not in fact cause the data size to shrink we 

836 # bail here because it's not worth trying to delete stuff from 

837 # the remainder. 

838 if lost_data <= 0: 

839 return False 

840 

841 # We now look for contiguous regions to delete that might help fix up 

842 # this failed shrink. We only look for contiguous regions of the right 

843 # lengths because doing anything more than that starts to get very 

844 # expensive. See minimize_individual_blocks for where we 

845 # try to be more aggressive. 

846 regions_to_delete = {(end, end + lost_data)} 

847 

848 for j in (blocks[-1] + 1, blocks[-1] + 2): 

849 if j >= min(len(initial_data.blocks), len(self.blocks)): 

850 continue 

851 # We look for a block very shortly after the last one that has 

852 # lost some of its size, and try to delete from the beginning so 

853 # that it retains the same integer value. This is a bit of a hyper 

854 # specific trick designed to make our integers() strategy shrink 

855 # well. 

856 r1, s1 = self.shrink_target.blocks[j].bounds 

857 r2, s2 = initial_data.blocks[j].bounds 

858 lost = (s1 - r1) - (s2 - r2) 

859 # Apparently a coverage bug? An assert False in the body of this 

860 # will reliably fail, but it shows up as uncovered. 

861 if lost <= 0 or r1 != r2: # pragma: no cover 

862 continue 

863 regions_to_delete.add((r1, r1 + lost)) 

864 

865 for ex in self.shrink_target.examples: 

866 if ex.start > start: 

867 continue 

868 if ex.end <= end: 

869 continue 

870 

871 replacement = initial_data.examples[ex.index] 

872 

873 in_original = [c for c in ex.children if c.start >= end] 

874 

875 in_replaced = [c for c in replacement.children if c.start >= end] 

876 

877 if len(in_replaced) >= len(in_original) or not in_replaced: 

878 continue 

879 

880 # We've found an example where some of the children went missing 

881 # as a result of this change, and just replacing it with the data 

882 # it would have had and removing the spillover didn't work. This 

883 # means that some of its children towards the right must be 

884 # important, so we try to arrange it so that it retains its 

885 # rightmost children instead of its leftmost. 

886 regions_to_delete.add( 

887 (in_original[0].start, in_original[-len(in_replaced)].start) 

888 ) 

889 

890 for u, v in sorted(regions_to_delete, key=lambda x: x[1] - x[0], reverse=True): 

891 try_with_deleted = bytearray(initial_attempt) 

892 del try_with_deleted[u:v] 

893 if self.incorporate_new_buffer(try_with_deleted): 

894 return True 

895 return False

897 def remove_discarded(self): 

898 """Try removing all bytes marked as discarded. 

899 

900 This is primarily to deal with data that has been ignored while 

901 doing rejection sampling - e.g. as a result of an integer range, or a 

902 filtered strategy. 

903 

904 Such data will also be handled by the adaptive_example_deletion pass, 

905 but that pass is necessarily more conservative and will try deleting 

906 each interval individually. The common case is that all data drawn and 

907 rejected can just be thrown away immediately in one block, so this pass 

908 will be much faster than trying each one individually when it works. 

909 

910 returns False if there is discarded data and removing it does not work, 

911 otherwise returns True. 

912 """ 

913 while self.shrink_target.has_discards: 

914 discarded = [] 

915 

916 for ex in self.shrink_target.examples: 

917 if ( 

918 ex.length > 0 

919 and ex.discarded 

920 and (not discarded or ex.start >= discarded[-1][-1]) 

921 ): 

922 discarded.append((ex.start, ex.end)) 

923 

924 # This can happen if we have discards but they are all of 

925 # zero length. This shouldn't happen very often so it's 

926 # faster to check for it here than at the point of example 

927 # generation. 

928 if not discarded: 

929 break 

930 

931 attempt = bytearray(self.shrink_target.buffer) 

932 for u, v in reversed(discarded): 

933 del attempt[u:v] 

934 

935 if not self.incorporate_new_buffer(attempt): 

936 return False 

937 return True

1295 # much, as it allows us to replace many of them in one go rather 

1296 # than one at a time. An example of where this matters is 

1297 # test_minimize_multiple_elements_in_silly_large_int_range_min_is_not_dupe 

1298 # in test_shrink_quality.py 

1299 def replace_range(k): 

1300 if k > c: 

1301 return False 

1302 

1303 def should_replace_byte(b): 

1304 return c - k <= b <= c and d < b 

1305 

1306 return self.consider_new_buffer( 

1307 hbytes([d if should_replace_byte(b) else b for b in buf]) 

1308 ) 

1309 

1310 find_integer(replace_range) 

1311 return True 

1312 

1313 if ( 

1314 # If we cannot replace the current byte with its predecessor, 

1315 # assume it is already minimal and continue on. This ensures 

1316 # we make no more than one call per distinct byte value in the 

1317 # event that no shrinks are possible here. 

1318 not can_replace_with(c - 1) 

1319 # We next try replacing with 0 or 1. If this works then 

1320 # there is nothing else to do here. 

1321 or can_replace_with(0) 

1322 or can_replace_with(1) 

1323 # Finally we try to replace with c - 2 before going on to the 

1324 # binary search so that in cases which were already nearly 

1325 # minimal we don't do log(n) extra work. 

1326 or not can_replace_with(c - 2) 

1327 ): 

1328 return 

1329 

1330 # Now binary search to find a small replacement. 

1331 

1332 # Invariant: We cannot replace with lo, we can replace with hi. 

1333 lo = 1 

1334 hi = c - 2 

1335 while lo + 1 < hi: 

1336 mid = (lo + hi) // 2 

1337 if can_replace_with(mid): 

1338 hi = mid 

1339 else: 

1340 lo = mid

1342 def run_block_program(self, i, description, original, repeats=1): 

1343 """Block programs are a mini-DSL for block rewriting, defined as a sequence 

1344 of commands that can be run at some index into the blocks 

1345 

1346 Commands are: 

1347 

1348 * "-", subtract one from this block. 

1349 * "X", delete this block 

1350 

1351 If a command does not apply (currently only because it's - on a zero 

1352 block) the block will be silently skipped over. 

1353 

1354 This method runs the block program in ``description`` at block index 

1355 ``i`` on the ConjectureData ``original``. If ``repeats > 1`` then it 

1356 will attempt to approximate the results of running it that many times. 

1357 

1358 Returns True if this successfully changes the underlying shrink target, 

1359 else False. 

1360 """ 

1361 if i + len(description) > len(original.blocks) or i < 0: 

1362 return False 

1363 attempt = bytearray(original.buffer) 

1364 for _ in hrange(repeats): 

1365 for k, d in reversed(list(enumerate(description))): 

1366 j = i + k 

1367 u, v = original.blocks[j].bounds 

1368 if v > len(attempt): 

1369 return False 

1370 if d == "-": 

1371 value = int_from_bytes(attempt[u:v]) 

1372 if value == 0: 

1373 return False 

1374 else: 

1375 attempt[u:v] = int_to_bytes(value - 1, v - u) 

1376 elif d == "X": 

1377 del attempt[u:v] 

1378 else: # pragma: no cover 

1379 assert False, "Unrecognised command %r" % (d,) 

1380 return self.incorporate_new_buffer(attempt)

1383def block_program(description): 

1384 """Mini-DSL for block rewriting. A sequence of commands that will be run 

1385 over all contiguous sequences of blocks of the description length in order. 

1386 Commands are: 

1387 

1388 * ".", keep this block unchanged 

1389 * "-", subtract one from this block. 

1390 * "0", replace this block with zero 

1391 * "X", delete this block 

1392 

1393 If a command does not apply (currently only because it's - on a zero 

1394 block) the block will be silently skipped over. As a side effect of 

1395 running a block program its score will be updated. 

1396 """ 

1397 name = "block_program(%r)" % (description,) 

1398 

1399 if name not in SHRINK_PASS_DEFINITIONS: 

1400 """Defines a shrink pass that runs the block program ``description`` 

1401 at every block index.""" 

1402 n = len(description) 

1403 

1404 def run(self, chooser): 

1405 """Adaptively attempt to run the block program at the current 

1406 index. If this successfully applies the block program ``k`` times 

1407 then this runs in ``O(log(k))`` test function calls.""" 

1408 i = chooser.choose(hrange(len(self.shrink_target.blocks) - n)) 

1409 # First, run the block program at the chosen index. If this fails, 

1410 # don't do any extra work, so that failure is as cheap as possible. 

1411 if not self.run_block_program(i, description, original=self.shrink_target): 

1412 return 

1413 

1414 # Because we run in a random order we will often find ourselves in the middle 

1415 # of a region where we could run the block program. We thus start by moving 

1416 # left to the beginning of that region if possible in order to to start from 

1417 # the beginning of that region. 

1418 def offset_left(k): 

1419 return i - k * n 

1420 

1421 i = offset_left( 

1422 find_integer( 

1423 lambda k: self.run_block_program( 

1424 offset_left(k), description, original=self.shrink_target 

1425 ) 

1426 ) 

1427 ) 

1428 

1429 original = self.shrink_target 

1430 

1431 # Now try to run the block program multiple times here. 

1432 find_integer( 

1433 lambda k: self.run_block_program( 

1434 i, description, original=original, repeats=k 

1435 ) 

1436 ) 

1437 

1438 run.__name__ = name 

1439 

1440 defines_shrink_pass()(run) 

1441 assert name in SHRINK_PASS_DEFINITIONS 

1442 return name

1458 def step(self): 

1459 if self.fixed_point_at is self.shrinker.shrink_target: 

1460 return False 

1461 

1462 tree = self.shrinker.shrink_pass_choice_trees[self] 

1463 if tree.exhausted: 

1464 return False 

1465 

1466 initial_shrinks = self.shrinker.shrinks 

1467 initial_calls = self.shrinker.calls 

1468 size = len(self.shrinker.shrink_target.buffer) 

1469 self.shrinker.explain_next_call_as(self.name) 

1470 try: 

1471 self.next_prefix = tree.step( 

1472 self.next_prefix, 

1473 lambda chooser: self.run_with_chooser(self.shrinker, chooser), 

1474 ) 

1475 finally: 

1476 self.calls += self.shrinker.calls - initial_calls 

1477 self.shrinks += self.shrinker.shrinks - initial_shrinks 

1478 self.deletions += size - len(self.shrinker.shrink_target.buffer) 

1479 self.shrinker.clear_call_explanation() 

1480 return True
